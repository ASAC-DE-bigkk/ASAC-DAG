# bronze — 영업상태 추적 모델 (단일 row vs 이력)

> 분석일 2026-06-30 · 실측 키 `sample` · 예시 `general_restaurant`(LOCALDATA_072404)
> 관련: [../common_info.md](../common_info.md) §2~3(공통 컬럼·UPDATEDT) · [pagination-ordering.md](pagination-ordering.md)

## 질문

> 한 업장의 영업상태 추적이 **하나의 row 안에서 모두** 이뤄지는가? 아니면 **상태 변경에 따라
> 컬럼(또는 row)이 추가**되며 동작하는가?

## 결론

**하나의 row(= 업장 1개, `MGTNO`)에서 모두 이뤄진다. 상태가 바뀌면 같은 row 의 고정 컬럼이
덮어쓰기(update)될 뿐, 컬럼도 row 도 추가되지 않는다.**

- LOCALDATA 응답은 **업장당 1행(현재 상태 스냅샷)**. `MGTNO`(관리번호)가 인허가 단위 식별자.
- 상태는 **고정된 컬럼 집합**으로 표현된다(아래 §evidence). 상태 변경 시:
  - **새 컬럼 추가 ✗** — 스키마는 고정(19 공통 컬럼 + 업종 고유 컬럼).
  - **새 row 추가 ✗** — 이벤트 로그/이력 테이블이 아니다.
  - **같은 row 갱신 ✓** — `TRDSTATEGBN`/`DCBYMD` 등이 최신 값으로 바뀌고,
    `LASTMODTS`(최종수정시점)·`UPDATEDT`(데이터갱신일자)·`UPDATEGBN`(I/U)이 갱신을 표시.
- 즉 이 API 는 **현재 상태(SCD Type 1, in-place overwrite)** 만 제공한다. 임의의 상태 전이
  이력은 **보존하지 않는다**.

## 단일 row 가 담는 "생애주기"

상태 변경 이력 대신, 한 row 는 생애주기 **마일스톤을 전용 날짜 컬럼**으로 담는다:

| 컬럼 | 의미 | 역할 |
|---|---|---|
| `APVPERMYMD` | 인허가일자 | 시작 마일스톤 |
| `CLGSTDT` / `CLGENDDT` | 휴업 시작 / 종료일 | 휴업 구간(식품군) |
| `APVCANCELYMD` | 인허가취소일자 | 취소 마일스톤 |
| `DCBYMD` | 폐업일자 | 종료 마일스톤 |
| `TRDSTATEGBN` / `TRDSTATENM` | 영업상태 코드 / 명 | **현재** 상태(01 영업·정상, 03 폐업 …) |
| `DTLSTATEGBN` / `DTLSTATENM` | 상세영업상태 | **현재** 상세 상태 |
| `UPDATEGBN` · `UPDATEDT` · `LASTMODTS` | 갱신구분 · 갱신일자 · 최종수정시점 | 변경 추적 메타 |

→ "얼마 동안 추적되나"는 곧 **인허가일 ~ 폐업/취소일의 생애 전체**지만, 그 사이의 모든 상태
전이가 아니라 **현재 상태 + 위 마일스톤 날짜**로만 압축된다. 예: 영업→휴업→영업→폐업을
거쳐도 휴업 컬럼(`CLGSTDT/CLGENDDT`)은 **마지막 값으로 덮어써져** 중간 이력은 사라진다.

## 증거 (general_restaurant 1~5행)

| MGTNO(축약) | TRDSTATE | DTLSTATE | 인허가 APVPERMYMD | 폐업 DCBYMD | 휴업 CLG | UPDATEGBN | UPDATEDT |
|---|---|---|---|---|---|---|---|
| …07985 | 03 폐업 | 02 폐업 | 2001-05-23 | **2007-02-07** | – | I | 2026-01-14 |
| …01852 | 03 폐업 | 02 폐업 | 1996-05-13 | **2008-09-29** | – | I | 2026-05-04 |
| …00202 | 01 영업/정상 | 01 영업 | 2007-08-27 | (없음) | – | I | 2026-01-14 |
| …02104 | 01 영업/정상 | 01 영업 | 1995-04-14 | (없음) | – | I | 2025-12-15 |
| …09890 | 03 폐업 | 02 폐업 | 2001-04-17 | **2002-04-19** | – | I | 2026-05-10 |

- 한 응답 안에서 **`MGTNO` 는 전부 distinct**(업장당 1행) — 같은 업장이 상태별로 여러 행으로
  나오지 않는다.
- **폐업 업장**: `TRDSTATEGBN=03` + `DCBYMD`(폐업일자) 채워짐. **영업 업장**: `01` + `DCBYMD` 빈값.
  즉 상태가 같은 row 의 값으로 표현되고, 폐업이라는 "변경"이 새 행/컬럼이 아니라 **값 갱신**으로
  반영됐다.
- `UPDATEDT`(2025~2026)는 오픈데이터 레코드가 갱신된 시점 — 같은 row 가 계속 **재갱신**됨을 보여준다.

## 파이프라인 함의

- API 가 이력을 주지 않으므로, **상태 변화 시계열은 우리가 직접 만들어야 한다.** bronze 가
  **run_id 폴더(실행시각)로 스냅샷을 누적**([../../architecture/storage.md](../../architecture/storage.md))하므로, 그 스냅샷들을
  비교해 **SCD Type 2(이력)** 를 silver/다운스트림에서 구성한다.
- 스냅샷 간 변경 감지 키: `MGTNO`(식별) + `TRDSTATEGBN`/`DTLSTATEGBN`(상태) +
  `LASTMODTS`/`UPDATEDT`(변경 시점). `UPDATEGBN`(I/U)로 신규/수정 구분
  ([../common_info.md](../common_info.md) §3).
- 같은 업장이 페이지 경계에서 흔들리거나 중복될 수 있으므로(정렬 키 부재,
  [pagination-ordering.md](pagination-ordering.md)) silver 는 **`MGTNO` 로 dedupe** 후 상태
  타임라인을 만든다.

> 요약: **추적은 1행 in-place**. 변경 = 컬럼/행 추가가 아니라 **그 행의 값 갱신**. 따라서
> "언제 영업→폐업했는지"의 단일 전이는 `DCBYMD`/`TRDSTATE`로 알 수 있어도, **연속적 상태
> 이력은 우리 스냅샷으로만** 얻는다.
